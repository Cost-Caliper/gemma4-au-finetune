# H11 Verdict — Fleet Phase 3 2x2 Specialist Benchmark

STATUS: FINAL. Ran live on Modal (`au-gemma4-phase3-specialists`, single A100-80GB,
2026-07-08/09), single seed, temperature 0. All numbers below are read directly from
`phase3/summary_<arm>_<set>.json` / `phase3/results_<arm>_<set>.jsonl`, generated by
`phase3/tool_eval.py` — nothing here is estimated or reconstructed from memory.

## Hypothesis (pre-registered)

H11: library-scoped LoRAs beat the whole-corpus generalist on their own library's tasks.

Decision rule: each specialist must (a) beat the CONTROL adapter on its own library's build
benchmark, AND (b) the 2x2 (specialists x libraries) must be diagonal. A null result (generalist
or retrieval-equivalent performance wins, no diagonal advantage) was pre-registered as a fully
acceptable, reportable outcome.

## Arms actually run (5/5)

1. `supabase` — phase1 supabase-scoped LoRA adapter, model name `supabase`
2. `vercel` — phase1 vercel-scoped LoRA adapter, model name `vercel`
3. `control` — phase1 whole-corpus generalist LoRA adapter (same recipe as 1/2), model name `control`
4. `base` — no adapter, model name `gemma4`
5. `v4` — EXPERIMENTAL Fireworks-format adapter (`/vol/adapter-v4`), model name `v4`. Going in,
   weight application was unproven (phase-0 spike: temp-0 output matched base verbatim on one
   simple prompt).

All 5 model names were verified live against the deployed endpoint before any eval spend
(`phase3/verify_models_result.json` — all 5 `"ok": true`, correct `model` field echoed back).

## Eval sets

- `phase1/eval_supabase.jsonl` — 15 build tasks, supabase curriculum (schema: question/reference/key_facts, split=code)
- `phase1/eval_vercel.jsonl` — 15 build tasks, vercel curriculum
- Supplementary (budget allowed, ran after builds): `phase1/eval_supabase_probes.jsonl` (69,
  on {supabase, control, base}) and `phase1/eval_vercel_probes.jsonl` (10, on {vercel, control,
  base})

## Scoring

- `key_fact_coverage` — fraction of reference key_facts (exact identifiers) present in the answer
- `judge_mean` — claude-haiku-4-5 0-10 rubric score vs. reference (API-correctness + task
  completeness), via `phase3/eval_common.py::judge()` (same rubric/logic as `../post_judge.py`)
- `tool_call_rate` — fraction of items where the model called `au_search` at least once (max 2
  rounds), parsed from vLLM's structured `tool_calls` field first, raw-text regex fallback second

### Errors (2/150 build-task attempts, both explained)

`control`/`eval_supabase` and `v4`/`eval_supabase` each had exactly 1 error (`HTTPError 400`),
both on the same duplicate-`level`-ID item (`L5-failure-modes-and-observability`, degree
06-supabase-vector, question ~6.2K chars). Reproduced live: the model called `au_search`, the
tool-result notes (up to 6 retrieved chunks x 2200 chars) pushed the round-2 prompt over
`--max-model-len 8192` when combined with the system prompt, tool schema, and `max_tokens=2400`.
This is a real, item-specific interaction between long tool-call payloads and the context budget
— it did not happen for `supabase`/`vercel`/`base` on the same item because those arms did not
choose to call the tool at all on it (see tool-call-rate finding below). Not a harness bug; not
patched around; reported as-is.

## Decision-rule evaluation

**(a) Does each specialist beat CONTROL on its own library's BUILD benchmark?**

- supabase adapter vs. control on `eval_supabase`: coverage 0.4390 > 0.4287 (specialist **wins**,
  +1.0pp) but judge 3.87 < 4.21 (specialist **loses**, −0.34). Mixed.
- vercel adapter vs. control on `eval_vercel`: coverage 0.4104 > 0.3616 (specialist **wins**,
  +4.9pp) but judge 3.60 < 3.93 (specialist **loses**, −0.33). Mixed.

Both specialists show a small coverage edge over control and a judge deficit vs. control, on
their own library. Neither metric alone, nor both together, gives a clean "specialist beats
control" result — and see the caveats below on why these margins are not distinguishable from
noise at n=15.

**(b) Is the 2x2 diagonal?** No. Two independent pieces of evidence rule this out:

1. **Every single arm — including `base` and `control`, which have no library affinity by
   construction — scores higher on `eval_supabase` than on `eval_vercel`**, on both coverage and
   (mostly) judge. That is a main effect of which eval set you're looking at (the supabase build
   tasks apparently have more literally-matchable identifiers), not an arm x library interaction.
   A true diagonal signal would show *specialists* elevated on their own set relative to *other
   arms'* pattern on that set — instead the whole matrix moves together by column.
2. **The vercel adapter's single best coverage result across the entire 2x2 is on the supabase
   set (0.4501)** — higher than its own vercel-set score (0.4104), and higher than the supabase
   adapter's own supabase-set score (0.4390). A specialist scoring best on someone else's library
   is the opposite of diagonal.

**Both legs of the pre-registered decision rule fail.** H11 is **not supported** by this run.

## Honesty check on magnitude

15 items per cell means one item's coverage classification flipping moves the cell average by
1/15 ≈ 6.7 percentage points. The largest specialist-over-control margin observed (vercel
adapter, +4.9pp coverage on its own set) is *smaller* than one item's worth of noise. None of the
build-set differences in this table should be read as a reliable effect at this sample size —
they are reported honestly, not because they clear a significance bar (there was none to clear;
none was pre-specified as a stat-sig threshold, and n=15/arm/set doesn't support one).

## A finding independent of the decision rule: tool-call rate is 100% arm-invariant

For every eval set, every one of the 5 arms called `au_search` on the exact same subset of items
(verified by comparing the per-item `id` lists, not just the aggregate rate): 8/15 items on
`eval_supabase` (tool_call_rate 0.533), 0/15 on `eval_vercel` (0.000), 0.986 on
`eval_supabase_probes`, 1.000 on `eval_vercel_probes` — identical, item-for-item, across
supabase/vercel/control/base/v4. Whether the model calls the tool on a given question is
apparently determined entirely by the question's content/structure at temperature 0 with this
fixed system prompt, not by which LoRA (or no LoRA) is loaded. Despite ~81-82% of the phase-1
training data being tool-call traces (per `phase1/training_report.md`), none of the 4 adapters
shifted tool-invocation propensity relative to base in this harness. This is a genuinely
surprising secondary finding, reported as observed — no causal explanation is claimed beyond
what the data shows.

## v4 arm validity call

Going in, the concern was: "if v4's results ≈ base, that confirms the adapter isn't applying."
What the data shows:

- v4 vs. base, `eval_supabase`: coverage 0.4323 vs 0.4496 (Δ −1.7pp), judge 3.71 vs 3.87 (Δ −0.16)
- v4 vs. base, `eval_vercel`: coverage 0.4033 vs 0.4107 (Δ −0.7pp), judge 3.73 vs 3.47 (Δ +0.26)

v4's aggregate scores do track base closely on both sets — consistent with the phase-0 concern.
**But this is not a clean confirmation**, for two reasons:

1. **The vercel adapter — a known-good, verified-training-recipe adapter (see
   `phase1/training_report.md`, `phase1/verify_report.json`) — is at least as close to base on
   coverage** (Δ +0.05pp on supabase-set, Δ −0.03pp on vercel-set — both smaller than v4's
   deltas). If "close to base" were sufficient to call an adapter a no-op, the vercel adapter
   would fail the same test, which we know is false (it loaded and trained successfully per
   phase-1 verification).
2. A spot check of raw answer text confirmed v4's outputs are **not byte-identical to base's**
   on any of the 9-10 items sampled (0/9, 0/10 exact matches) — same as every other arm compared
   to base. So v4 is not a literal no-op at the API/text layer; whether its LoRA weights are
   contributing a *behaviorally meaningful* delta beyond base is what's actually in question, and
   this benchmark's sample size cannot resolve that for v4 any more than it can for the other
   three adapters, which show similarly small deltas from base and from each other.

**Call: INCONCLUSIVE, not "confirmed invalid."** The result is consistent with the pre-registered
suspicion but does not clear the bar the brief itself set (results "matched base" — they're close
but not statistically distinguishable, and neither are the verified-good adapters at this n). The
honest statement is: **at n=15/cell, none of the four adapters (supabase, vercel, control, v4)
produce build-task scores distinguishable from base or from each other** — v4 is not uniquely
suspicious in this data, it just happens to be the one arm we had prior reason to doubt.

## Caveats

- Single seed, temperature 0, 15 items per cell on the build sets — no confidence intervals; a
  single item's classification flip moves a cell by ~6.7 percentage points (coverage) or up to
  10/n in judge-scale terms.
- Judge is claude-haiku-4-5 (LLM-judge noise, not a ground-truth oracle); 3 build cells lost 1
  judge score each to a JSON-parse failure on the judge call itself.
- `eval_supabase.jsonl` has at least one duplicate `level` id across two different degrees
  (`L5-failure-modes-and-observability` appears for both `05-supabase-realtime` and
  `06-supabase-vector`) — noted because it caused ambiguity when spot-checking by id; it does not
  affect per-item scoring (each record is scored independently), only cross-referencing by id.
- Retrieval quality (local AU API, port 3030, same corpus/index for every arm) was held identical
  across all arms — differences in tool-call outcomes reflect the model's choice of query and use
  of results, not differences in what was retrievable.
- Probe sets are supplementary evidence, not part of the pre-registered decision rule, and were
  not run for the v4 arm (budget was prioritized toward supabase probes on {supabase, control,
  base} per the brief) or for the supabase adapter on vercel probes / vercel adapter on supabase
  probes (out of scope for their own-library check).
- 2 of 387 total item attempts (0.5%) errored on context-length overflow, both explained above and
  reported, not silently dropped or patched around.

## Verdict (honest)

**H11 is not supported by this benchmark.** Neither the supabase-scoped nor the vercel-scoped
LoRA reliably beats the whole-corpus generalist control adapter on its own library's build tasks
when both coverage and judge metrics are considered — each specialist wins on coverage and loses
on judge, by margins smaller than the noise floor at this sample size. The 2x2 is clearly not
diagonal: every arm (specialist or not) scores better on the supabase build set than the vercel
build set, a property of the eval sets themselves, not of specialization — and the vercel
adapter's best result in the whole matrix is on someone else's library. Tool-call behavior was
100% determined by question content and identical across all 5 arms, suggesting the LoRA
fine-tune (attention-only, rank 32, per `phase1/training_report.md`) did not measurably shift
tool-use propensity at this training scale. The v4 experimental arm's results track base closely,
consistent with (but not proof of) its suspected non-application — the same "close to base"
pattern is also true of the two verified-good specialist adapters at this sample size, so v4
cannot be singled out as confirmed-inert from this data alone.

This is the honest null the brief pre-registered as an acceptable outcome: **the generalist
control was, if anything, competitive-to-ahead of the specialists on the judge metric, and no
diagonal specialization advantage was detected.** A larger build-task n per cell (the probe sets
hint the picture doesn't clean up much even at n=69/n=10) would be needed to distinguish a real
small effect from noise, if one exists.
