## Cause

Supabase configures the `public` schema with default GRANTs to the `anon` and `authenticated` roles. The combination of:

- `public` schema is **exposed via PostgREST** (default in `config.toml`)
- `anon` has `SELECT, INSERT, UPDATE, DELETE` GRANTs by default on `public.*`
- RLS is **NOT** auto-enabled (gotcha-rls-not-auto-enabled.md)

…means a `CREATE TABLE public.x` is, by default, **wide open to anyone with the anon key**.

<start_of_turn>model
## If Asked to Write an RLS Policy

Template (substitute `<table>` and `<user_col>`):

```sql
-- SELECT: read own rows
drop policy if exists "<table>_select_own" on public.<table>;
create policy "<table>_select_own"
  on public.<table> for select to authenticated
  using ( (select auth.uid()) = <user_col> );

-- INSERT: create only own rows
drop policy if exists "<table>_insert_own" on public.<table>;
create policy "<table>_insert_own"
  on public.<table> for insert to authenticated
  with check ( (select auth.uid()) = <user_col> );

-- UPDATE: USING for finding the row, WITH CHECK for the new value
drop policy if exists "<table>_update_own" on public.<table>;
create policy "<table>_update_own"
  on public.<table> for update to authenticated
  using ( (select auth.uid()) = <user_col> )
  with check ( (select auth.uid()) = <user_col> );

-- DELETE: USING only
drop policy if exists "<table>_delete_own" on public.<table>;
create policy "<table>_delete_own"
  on public.<table> for delete to authenticated
  using ( (select auth.uid()) = <user_col> );
```

**Mandatory checks before committing:**

- [ ] Used `(select auth.uid())` not bare `auth.uid()`
- [ ] Specified `to authenticated` (or the correct target role) on every policy
- [ ] UPDATE policy has BOTH `using` and `with check`
- [ ] INSERT policy has `with check` (not `using`)
- [ ] A SELECT policy exists if there is an UPDATE policy
- [ ] FK column referenced in the policy has an index
- [ ] RLS is enabled on the table

Checklist: `checklists/checklist-rls-policy-review.md`

What did hands-on work with supabase teach about: Lesson 01 — Supabase Auth Mental Model?<end_of_turn>
<start_of_turn>model
## The Mental Picture

When you call `signInWithPassword`:

1. supabase-js POSTs `{email, password}` to `GoTrue /auth/v1/token?grant_type=password`.
2. GoTrue validates against the `auth.users` row (bcrypt over `encrypted_password`), then:
   - If valid → signs and returns `{access_token, refresh_token, expires_in, ...}`.
   - If invalid (wrong pw OR unknown email) → returns `{code:'invalid_credentials', status:400}`.
3. supabase-js stores the pair in `localStorage` (default `storage`), emits `SIGNED_IN` on `onAuthStateChange`.
4. Subsequent PostgREST calls inject `Authorization: Bearer <at>`.
5. PostgREST hands the JWT to Postgres, which sets `request.jwt.claims` GUC and `SET ROLE <claim.role>`. RLS policies see `auth.uid()` returning the `sub`.

Internalising this five-step path means you can debug any auth bug by asking: where in this chain did the contract break?
